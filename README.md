# 开发文档管理 — Claude Skill

让 Claude Code 自动管理 PRD、Changelog、Dev-Log、Todo List，把产品定义和开发过程用文档沉淀下来，不再靠对话记录回忆。

## 要解决什么问题

| 问题 | 后果 | 谁解决 |
|------|------|--------|
| 功能边界模糊 — 需求没对齐就开写，交互细节和技术约束都是糊涂账 | 写到一半发现理解不一致，反复改，时间全耗在返工上 | **PRD** |
| 变更记录缺失 — 改了什么、为什么改，没有系统记录 | 复盘全靠回忆，一周后能想起一半都算不错 | **Changelog** |
| 开发过程丢失 — 踩坑、选型原因、决策上下文，对话结束就沉底了 | 未来自己打开项目像看没有注释的代码，同样问题重新踩一遍 | **Dev-Log** |
| 需求散落各处 — "放到 v0.5 再做"的约定丢在聊天窗口里 | 下一次想起来已经是几个月后，或者彻底忘了 | **Todo List** |

## 怎么解决

**核心思想：先写 PRD，审阅通过，再开发。** 在写任何代码之前，AI 必须先输出一份完整的 PRD，由你逐条确认功能范围、交互方式、技术方案，对齐后再开始编码。避免「边写边想、边改边猜」的混乱。

在此基础上，用一套标准化的文档体系管理全流程信息：

| 文档 | 记录什么 | 更新时机 |
|------|----------|----------|
| **PRD** | 产品定义、功能规格、技术方案 | 功能版本升级时 |
| **Changelog** | 每次交付改了什么、为什么改（三句话） | 每次有可交付结果时同步 |
| **Dev-Log** | 开发过程日记（踩坑、决策、尝试） | 开发过程中持续记录 |
| **Todo List** | 待办任务，按优先级分层 | 用户手动维护，AI 按需协助 |

## 安装

```bash
# skill 已放入 ~/.claude/skills/ 即可
# 可选：安装 git hook，commit 后提醒更新文档
cp hook-settings.local.json 你的项目/.claude/settings.local.json
```

验证生效：执行一次 `git commit`，如果看到提示：

```
📋 提示：提交前请确认已同步 Dev-Log / Changelog / PRD。
```

则安装成功。

## 使用方式

本 skill **自动触发**，不需要手动调用。只要满足以下场景之一，AI 会自动执行文档管理 SOP：

- 你提出开发新产品、新增/修改/删除功能需求
- AI 实际修改了代码、配置、脚本、测试、文档
- 你说「写 Changelog」「归档」「整理文档」「更新一下」

### 你只需要做的事

1. 审阅 AI 写的 PRD，确认功能范围
2. 正常写代码、跟 AI 讨论方案
3. AI 会在每次对话结束时自动更新 Dev-Log / Changelog / PRD

### 新产品场景

当你说「我想开发一个新产品」或「帮我做一个 XX 应用」时，AI 会自动：

1. 创建产品总目录（如 `20260603 智能笔记/`）
2. 创建 4 份初始文档
3. 编写 PRD 草案供你审阅
4. 审阅通过后再开始编码

### 已有项目场景

AI 会自动定位已有的 PRD 文件，判断本轮改动类型：

- **修了个 bug / 做了个小调整** → 追加 Dev-Log
- **交付了新功能** → 追加 Dev-Log + Changelog
- **用户可感知的功能变化** → PRD 升版 + 同步更新 Changelog
- **你说「放到 v0.5」「下个版本做」** → 写入 Todo List + Dev-Log

## 文档结构

```
📁 {产品总目录}/                          ← 如 20260531 AI 伴读/
  📄 {日期} PRD-v{版本}-最新-{产品名}.md   ← 唯一当前生效文件
  📄 {日期} Changelog-{产品名}.md           ← 版本变更记录
  📄 {日期} Dev-Log-{产品名}.md             ← 开发过程日记
  📄 {日期} Todo-List-{产品名}.md           ← 待办清单
  📁 Archive/
    📄 {旧日期} PRD-v{旧版本}-{产品名}.md    ← 旧版归档

📁 {代码项目目录}/                         ← 如 reading-assistant/
  📁 src/
  ...
```

### 关键规则

- 根目录下**只能有一份** `最新` 的 PRD 文件
- 旧版归档时移入 `Archive/`，文件名去掉 `-最新` 标识
- 活跃 PRD、Changelog、Dev-Log、Todo 的日期前缀必须一致
- 跨天继续开发时，所有活跃文档自动重命名为当天日期前缀

## PRD 版本判断

| 条件 | 是否升版 |
|------|----------|
| 新增/删除功能模块 | ✅ 升次版本 |
| 核心交互变更 | ✅ 升次版本 |
| 存储/架构变更 | ✅ 升次版本 |
| 产品定位变化 | ✅ 升主版本 |
| 错别字、文案修正 | ❌ 直接改当前文件 |
| UI 描述细节调整 | ❌ 直接改当前文件 |

## 示例

首次使用时，AI 会执行初始化检查：

```
当前生效 PRD：260531 PRD-v0.5-最新-AI-伴读.md
最新版本：v0.5
归档文件：（无）
Changelog：260606 Changelog-AI-伴读.md
Dev-Log：260606 Dev-Log-AI-伴读.md
Todo-List：260606 Todo-List-AI-伴读.md
```

## 设计原则

- **PRD 先行**：功能边界在写代码之前对齐，减少返工
- **Changelog 极简**：默认只写「问题 / 原因 / 解决」三句话，详细过程放 Dev-Log
- **Dev-Log 事无巨细**：踩坑、尝试、决策上下文都记，不怕啰嗦
- **Todo 用户主导**：AI 不随意加任务，只在你明确要求或完成已有任务时协助更新
- **版本号不可逆**：旧版归档而非覆盖，版本只能升不能降
