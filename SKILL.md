## 一、概述

### 1.1 要解决什么问题

开发过程中经常出现以下混乱：

1. **功能边界模糊**：直接跟 AI 说需求就开写，即使用了 plan 模式，功能边界、交互细节、技术约束仍然没有说清楚。写到一半发现理解不一致，反复修改浪费时间
2. **变更记录缺失**：改了什么功能、为什么改，没有系统记录，复盘时全靠回忆
3. **开发过程丢失**：踩了什么坑、试了什么方案、为什么做某个技术决策，这些上下文在对话结束后就丢了
4. **需求散落各处**："放到下个版本做""之后再处理"的约定散落在对话里，没有统一记录，下一次想起来已经是几个月后

### 1.2 解决方式

**核心思想：先写 PRD，审阅通过，再开发。**

在开发任何产品之前，AI 必须先输出一份完整的 PRD，由用户逐条审阅确认功能范围、交互方式、技术方案等细节，双方对齐后再开始编码。这样避免「边写边想、边改边猜」的混乱，让自然语言沟通中的模糊地带在写代码之前就被澄清。

在此基础上，本 skill 用一套标准化的文档体系来管理开发全流程的信息：

- **PRD** 记录产品定义——只有一份当前生效，旧版自动归档，避免版本混乱
- **Changelog** 记录变更摘要——每次版本变更同步更新，快速回溯版本差异
- **Dev-Log** 记录开发过程——踩坑、决策、讨论随时记录，保留上下文
- **Todo List** 记录任务规划——待办事项按轻重缓急分层管理，手动维护

四类文档各有分工，配合一套完整的命名规则和执行流程，让开发文档始终处于"打开即用"的状态。

## 二、文档体系

### 2.1 四类文档的分工

| 维度               | PRD（产品需求文档） | Changelog（变更记录） | Dev-Log（开发日记）    |
| ------------------ | ------------------- | --------------------- | ---------------------- |
| **记录什么** | 产品定义、功能规格  | 版本间的变化摘要      | 开发过程、决策上下文   |
| **给谁看**   | 团队对齐产品方向    | 快速回顾版本差异      | 未来的自己理解当时决策 |
| **更新时机** | 功能版本升级时      | 每次版本变更时同步    | 开发过程中持续记录     |
| **详细程度** | 完整功能描述        | 一行结论 + 原因       | 1-2 句上下文           |
| **文件数量** | 根目录仅一份最新    | 单个持续追加文件      | 单个持续追加文件       |

### 2.2 文件命名与归档规则（强制）

**PRD 命名格式：**

```
{YYYYMMDD} PRD-v{主版本}.{次版本}-{状态}-{产品名}.md
```

- `YYYYMMDD`：定稿或创建日期，如 `260531`
- `v{主版本}.{次版本}`：
  - `v0.x` = 开发中 / MVP（遵循语义化版本规范，v0 阶段表示产品尚未稳定，功能快速迭代中）
  - `v1.0+` = 正式发布
  - 每次功能变更次版本 +1，重大重构主版本 +1
- `{状态}`：二选一
  - `最新` = 当前唯一生效版本
  - `草案` = 正在编写，尚未生效
- `{产品名}`：如 `Apple-Books-AI-阅读助手`

**Changelog 命名格式：**

```
{YYYYMMDD} Changelog-{产品名}.md
```

- 无版本号，只有一个持续追加的记录文件
- 顶层 Changelog 放在产品总目录（与当前 `最新` PRD、Dev-Log 同级）
- 不在代码项目目录中维护第二份 Changelog；如果发现历史遗留的代码级 Changelog，应先把内容合并到产品根目录 Changelog，再将代码目录副本移入产品根目录 `Archive/` 或删除，避免文档分叉
- 每次 PRD 版本更新时同步追加
- 活跃 Changelog、Dev-Log、Todo List、测试清单的文件名前缀必须与当前最新 PRD 的日期一致；如果创建了新的最新 PRD 或当天发生版本级变更，必须把仍在维护的开发文档重命名到同一 `YYYYMMDD` 前缀。旧版 PRD 才放 Archive，活跃开发文档不要停留在旧日期。
- **跨天硬规则**：只要今天继续修改当前项目的代码、配置、测试清单或开发文档，且当前活跃 PRD/Changelog/Dev-Log/Todo/测试清单仍是昨天或更早日期，必须在写入任何新内容前先把所有活跃文档统一重命名为今天的 `YYMMDD` 前缀，包括当前 `最新` PRD。本规则即使没有升版本也必须执行。
- **正文日期同步硬规则**：重命名活跃文档后，必须同步更新文档正文中的当前生效日期：PRD 头部 `**日期**`、Changelog 当前顶部版本节日期、Dev-Log 当日章节、Todo 当前完成日期/当日整理记录、测试清单标题或引用中的当前文件名日期。不能只改文件名而让正文元数据停留在旧日期。

**Dev-Log 命名格式：**

```
{YYYYMMDD} Dev-Log-{产品名}.md
```

**Todo List 命名格式：**

```
{YYYYMMDD} Todo-List-{产品名}.md
```

- 无版本号，用户手动维护
- 放在产品总目录（与当前 `最新` PRD、Changelog、Dev-Log 同级）
- 默认不主动新增无关 Todo，但以下情况必须更新 Todo：用户提出新的开发/修复/优化需求且本轮需要处理；用户明确要求写入/整理/删除 Todo；用户指定未来版本需求，如“放到 v0.5”“之后再处理”；本轮实现完成了 Todo 中已有条目，需要标记完成并移入「已完成」区。

### 2.3 文件夹结构

```
📁 {产品总目录}/                          ← 如 20260531 AI 伴读/
  📄 {最新日期} PRD-v{版本}-最新-{产品名}.md   ← 唯一当前生效文件
  📄 {日期} Changelog-{产品名}.md               ← 版本变更记录（顶层）
  📄 {日期} Dev-Log-{产品名}.md                 ← 开发过程日记
  📄 {日期} Todo-List-{产品名}.md                 ← 待办清单（手动维护）
  📁 Archive/
    📄 {旧日期} PRD-v{旧版本}-{产品名}.md        ← 原最新文件，去掉"最新"状态

📁 {代码项目目录}/                       ← 如 reading-assistant/
  📁 src/
  📁 src-tauri/
  ...
```

### 2.4 核心规则

1. 根目录下**只能有一份** `最新` 的 PRD 文件
2. 旧版归档时移入 `Archive/` 即可，文件名不变，去掉 `-最新` 标识
3. 不允许存在无状态标识的文件名
4. 如果当前工作目录在代码子目录内，必须向上查找产品总目录，优先使用包含 `*PRD-v*-最新-*`、`*Changelog-*`、`*Dev-Log-*` 的目录作为文档根目录
5. 当前 `最新` PRD 日期与活跃 Dev-Log/Changelog/Todo List/测试清单日期不一致时，先同步重命名活跃开发文档，再继续写内容。
6. 如果当前日期已经晚于活跃文档文件名前缀日期，且本轮会修改项目或开发文档，先将当前 `最新` PRD、Changelog、Dev-Log、Todo List、测试清单全部重命名为今天 `YYMMDD` 前缀，再同步正文中的当前生效日期字段/顶部章节日期/当前引用日期，最后写入任何新记录。

### 2.5 待办事项（Todo List）

#### 定位

Todo List 是**任务规划清单**，记录「之后需要做但还没做的事」。默认由用户规划维护，AI 不应随意新增无关任务；但用户提出开发/修复/优化需求、明确要求、指定未来版本需求，或完成已有 Todo 条目时，AI 必须同步更新。

#### 推荐结构

建议按「完成状态 → 优先级」分层组织：

```
# Todo List - {产品名}

## 未完成

### 🔴 高优先级
- [ ] 任务描述

### 🟡 中优先级
- [ ] 任务描述

### 🟢 低优先级
- [ ] 任务描述

## 已完成
- [x] 已完成的任务描述（完成日期：YYYY-MM-DD；结果：一句话说明） ✅
```

#### 核心规则

- Todo List 默认由用户规划维护，AI 不应随意新增无关任务
- **需求入列硬规则**：用户提出任何需要修改产品、代码、配置、脚本、测试、文档或体验的任务时，应先确认它是否属于当前项目 Todo；属于则必须写入「未完成」区。若本轮立即完成，不必长期停留在未完成区，但最终回复前必须移入「已完成」区并加 `✅`
- 用户明确要求写入 Todo、整理 Todo、标记完成，或明确指定未来版本需求时，AI 必须更新 Todo
- **完成收尾硬规则**：如果本轮实现、修复、验证或交付了 Todo List「未完成」区中的任务，必须在最终回复前同步更新 Todo List：将对应 `- [ ]` 条目从「未完成」区移除，改为 `- [x]` 后放入「已完成」区，并在条目末尾加 `✅`；如果没有「已完成」区，必须自动创建。
- 已完成条目应尽量保留原任务描述，并补充完成日期与结果摘要，末尾必须有醒目的 `✅`；不要只在 Changelog/Dev-Log 记录完成而让 Todo 继续停留在「未完成」。
- 未完成条目可以继续按优先级保留；已完成条目可定期归档或清理，但不得在刚完成时直接删除到不可追踪。

## 三、Skill 运作 SOP

本 SOP 定义了每次对话结束时 AI 必须执行的文档管理操作。只要本轮对项目有任何改动（代码、配置、脚本、测试、文档等），在最终回复用户之前必须走完这个流程。

### 3.1 步骤一：定位文档根目录

1. 从当前工作目录开始，逐级向上查找以下文件：
   - `*PRD-v*-最新-*.md`
   - `*Dev-Log-*.md`
   - `*Changelog-*.md`
2. 找到当前 `最新` PRD 的目录即为产品文档根目录
3. 如果当前目录是代码目录（如 `reading-assistant/`），不要把代码目录误当成文档根目录——除非那里也有当前 `最新` PRD
4. 如果存在多份 Dev-Log 或 Changelog，优先选择与当前 `最新` PRD 同级、产品名一致、日期最新的文件
5. 如果 `最新` PRD、Changelog、Dev-Log、Todo List、测试清单的活跃文件日期前缀不一致，必须先重命名到当前最新 PRD 的日期前缀
6. 如果今天日期晚于当前最新 PRD 文件名前缀日期，且本轮将继续改项目或写开发文档，必须先把当前最新 PRD 和所有活跃开发文档一起重命名到今天 `YYMMDD` 前缀
7. 重命名后必须检查并更新正文日期：PRD 头部 `**日期**` 应为今天 `YYYY-MM-DD`；Changelog 顶部当前版本节应为今天 `YYYYMMDD`；Dev-Log 应有今天 `YYYY-MM-DD` 章节；测试清单和 Todo 中的当前活跃引用不得继续指向旧日期文件名。
6. 如果以上文件均未找到，说明这是**全新的产品项目**：
   a. 与用户确认产品名和工作目录
   b. 在工作目录下创建产品总目录（以日期+产品名命名，如 `20260603 智能笔记/`）
   c. 在该目录下创建 4 份初始文档（见下方「新产品初始化清单」），其中 PRD 编写完整草案内容
   d. 将 PRD 草案提交用户审阅，确认后再进入开发阶段

> **新产品初始化清单：**
> - 创建 `{今日日期} PRD-v0.1-草案-{产品名}.md`，填写完整的产品定义、功能规格、技术方案
> - 创建 `{今日日期} Dev-Log-{产品名}.md`，写入当日首条记录
> - 创建 `{今日日期} Changelog-{产品名}.md`，写入初始版本记录（v0.1）
> - 创建 `{今日日期} Todo-List-{产品名}.md`，按推荐结构生成空白模板

### 3.2 步骤二：判断本轮改动类型

检查本轮实际做过的事，判断需要更新哪些文档：

| 条件                                                                            | 需执行的操作                                      |
| ------------------------------------------------------------------------------- | ------------------------------------------------- |
| **用户提出开发新产品**                                                          | → **新产品初始化：创建 4 份文档 + 编写 PRD 草案（见 3.1 第 5 步）** |
| 有技术尝试、踩坑、测试、实现路径、约束变化、用户重要决定                        | →**追加 Dev-Log**                          |
| 交付结果改变了功能、行为、配置、构建方式、依赖、用户可感知体验                  | →**追加 Dev-Log + Changelog**              |
| 用户可感知的功能模块、核心交互、存储/架构、产品定位、UI/UX 大改、技术栈发生变化 | →**完整 PRD 升版（见 3.4 分支 C）**        |
| 纯排版、错别字、说明文字微调                                                    | → 可只写 Dev-Log；若用户明确要求，也写 Changelog |
| 用户指定未来版本需求（"放到 vX.Y""下个版本做"）                                 | → 写入当前 PRD 的未来计划，同步 Todo List、Dev-Log    |
| 用户指出开发文档日期/命名不一致                                                   | → 先重命名活跃开发文档，再更新 skill 规则或 Dev-Log    |

> **注意**：Todo List 默认不自动乱写；但用户明确要求、未来版本规划、或完成已有 Todo 时，必须同步更新。

### 3.3 步骤三：执行文档更新

根据判断结果，走对应分支：

---

#### 分支 A：仅追加 Dev-Log

在 `Dev-Log` 文件中追加记录。不要只写"已修改"，必须写清过程。

推荐格式：

```
## {YYYY-MM-DD}

- {类型}：{本轮做了什么，涉及哪些文件/模块，为什么这样做}
- {过程}：{关键尝试、踩坑、替代方案、取舍；没有踩坑也写验证路径}
- {结果}：{最终状态、剩余风险、后续待办}
```

类型可用：`需求`、`实现`、`修复`、`调试`、`技术决策`、`踩坑`、`文档`、`验证`。

---

#### 分支 B：追加 Dev-Log + Changelog

先按分支 A 追加 Dev-Log，然后在 `Changelog` 文件顶部追加一节：

```
## {YYYYMMDD} — {一句话摘要}

- 问题：{发现了什么问题或本次交付改变了什么}
- 原因：{为什么会有这个问题，或为什么需要这个变更}
- 解决：{本轮用什么方式解决，保持一句话}
```

**Changelog 书写规则：**

- 默认只写三句话：`问题`、`原因`、`解决`
- 每句话尽量一行，不写实现细节、文件清单、完整验证过程
- 详细调试过程、踩坑、替代方案、验证命令全部写入 Dev-Log
- 日期使用 `YYYYMMDD`

---

#### 分支 C：完整 PRD 升版

分支 C 走以下 8 个子步骤：

**子步骤 0：扫描与确认**

1. 列出根目录所有 `PRD-*` 文件
2. 确认当前 `最新` 文件及其版本号
3. 读取当前 `Changelog-*` 和 `Dev-Log-*` 的最新版本号

**子步骤 1：判断升级类型**

- 根据「版本升级判断标准」（见第四章）确定升级类型
- 用户可感知的功能变化 → 次版本 +1
- 细节调整 → 修订号 +1

**子步骤 2：创建新版 PRD**

1. 复制当前 `最新` 文件
2. 新文件名：`{今日日期} PRD-v{新版本}-草案-{产品名}.md`
3. 在文件头部添加：
   ```
   版本：v{x.y}
   日期：{YYYYMMDD}
   状态：草案
   变更摘要：{一句话概括本次变更}
   ```

**子步骤 3：执行内容修改**

- 按用户要求修改 PRD 内容
- 确保结构完整（定位、背景、功能、技术方案、开发计划）

**子步骤 4：同步更新 Changelog**

- 参照分支 B 的 Changelog 格式，在顶部新增一节
- 版本号标记为此次的新版本号

**子步骤 5：定稿与归档**

1. 将新版重命名为 `最新`
2. 将旧版 `最新` 移入 `Archive/`
3. 旧版文件名中去掉 `-最新`，保持其他部分不变
4. 确认根目录下只有一份 `最新` 文件

**子步骤 6：清理代码级 Changelog（如有历史遗留）**

如果代码项目目录内存在 `{日期} Changelog-{产品名}.md`，不要继续同步维护第二份。必须先确认其中没有顶层 Changelog 缺失的重要内容；如有，合并到产品根目录 Changelog；随后将代码目录副本移入产品根目录 `Archive/` 或删除，避免用户在 Finder 中误以为 Changelog 不在根目录。

**子步骤 7：验证**

- [ ] 根目录 `最新` 文件数量 = 1
- [ ] 旧版已移入 `Archive/` 且去掉了 `-最新` 标识
- [ ] `Changelog` 已同步更新且格式正确
- [ ] 版本号连续（无跳跃）

### 3.4 步骤四：最终回复前自检

最终回复用户前必须确认：

- [ ] 已定位当前 `最新` PRD、Changelog、Dev-Log
- [ ] 本轮有项目改动时，Dev-Log 已记录
- [ ] 本轮有可交付结果时，Changelog 已记录
- [ ] 本轮涉及用户可感知功能变化时，PRD 已升版，或已说明为什么不升版
- [ ] 本轮如果用户指定未来版本需求，PRD 未来版本计划、Todo List 和 Dev-Log 都已同步
- [ ] 本轮如果完成了 Todo List「未完成」区中的任务，已将对应条目改为 `[x]`、移入「已完成」区，并在末尾加 `✅`
- [ ] 当前最新 PRD、Changelog、Dev-Log、Todo List、测试清单的活跃文件日期前缀一致
- [ ] 如果今天已经不是活跃文档前缀日期，已先把当前最新 PRD、Changelog、Dev-Log、Todo List、测试清单统一重命名为今天 `YYMMDD` 前缀
- [ ] 活跃文档正文中的当前生效日期也已同步：PRD 头部日期、Changelog 顶部当前版本日期、Dev-Log 当日章节、Todo/测试清单当前引用日期均未停留在旧日期
- [ ] 回复里简要说明写入了哪些开发文档

如果因为路径冲突、多份最新 PRD、权限问题等无法写入，必须在最终回复中明确说明阻塞原因和待用户确认事项。

## 四、PRD 升级判断标准

版本号格式为 `v{主}.{次}`（只有两位）。`v0.x` 阶段表示产品正在开发中，每次功能变更次版本 +1。正式发布后进入 `v1.0+`。

### 必须升级版本（任意一条满足即升）

| # | 条件                             | 示例                                               |
| - | -------------------------------- | -------------------------------------------------- |
| 1 | **新增/删除功能模块**      | 新增 F4 导入书籍、删除 F3 云同步                   |
| 2 | **核心交互变更**           | 快捷键从 ⌥+S 改为 ⌥+A；触发方式从快捷键改为双击  |
| 3 | **存储/架构变更**          | 从 SQLite 改为 Markdown 文件；从本地存储改为云同步 |
| 4 | **产品定位或目标用户变更** | 从"个人阅读工具"改为"团队协作平台"                 |
| 5 | **UI/UX 重大改版**         | 整体风格重设计、窗口模式改为浮动面板               |
| 6 | **技术栈更换**             | 从 Tauri 改为 Electron；从 React 改为 Vue          |

### 不需要升级版本（只改内容，不改文件）

| # | 条件             | 示例                               |
| - | ---------------- | ---------------------------------- |
| 1 | 文案修正、错别字 | 修改错字、调整语句通顺             |
| 2 | UI 描述细节调整  | 按钮位置描述、颜色描述调整         |
| 3 | 预设提示词增删   | 新增"翻译"按钮、删除不常用的"总结" |
| 4 | 格式美化         | 调整表格样式、补充空行、修正标点   |
| 5 | 纯技术实现细节   | 改用哪个 Rust crate、API 调用方式  |

### 判断流程

```
用户要求改 PRD
    │
    ├─ 满足"必须升级"任意一条？──→ 次版本 +1（v0.x 阶段 → v0.N+1，v1.0+ 阶段 → v1.N+1），走完整升版流程
    │
    └─ 仅"不需要升级"的范围？──→ 直接改当前最新文件，不建新版本
```

**核心原则**：任何用户能感知到的功能变化 → 升版本。纯文字/格式修饰 → 不升。

## 五、Dev-Log 记录规范

### 定位

Dev-Log 是**过程日记**，记录「为什么改、踩了什么坑、试了什么方案、当时怎么想的」。放在项目文件夹内，作为开发过程的第一手记录。

### 记录时机（重要）

**在整个对话过程中持续记录**，不等到版本更新才写。只要本轮对项目文件有改动，最终回复前必须追加。以下场景必须立即追加：

1. **用户提问技术问题** → 记录问题和你的回答/建议
2. **踩坑/解决方案** → 记录遇到了什么坑、怎么解决的
3. **技术选型讨论** → 记录 A 方案和 B 方案的权衡
4. **用户提出新想法** → 即使未实现也记录，以后可能用上
5. **每次代码/配置/脚本/测试/文档变更后** → 记录实际改动、验证方式、剩余风险
6. **每次对话结束时** → 检查当天有没有遗漏未记录的讨论

### 文件位置

```
📁 {项目根目录}/
  📄 {YYYYMMDD} Dev-Log-{产品名}.md
```

### 格式

```
## {YYYY-MM-DD}
- {决策/踩坑/放弃/灵光一现，1-2句话}

## {YYYY-MM-DD}
- {技术选型尝试：试了 A 库发现不行，改用 B，原因：...}
- {重要决策上下文：为什么改快捷键、为什么删功能}
```

### 记录标准（满足任意一条就记）

- 技术选型尝试与放弃
- 重要决策的上下文
- 开发效率/情绪状态
- 未实现的想法（灵光一现，留待验证）

## 六、Changelog 记录规范

### 定位

Changelog 是**变更记录**，记录「每个版本或每次交付改了什么、为什么改」。放在产品总目录内，作为版本差异的快速查阅入口。

### 与 Dev-Log 的分工

| 维度     | Changelog                   | Dev-Log                                    |
| -------- | --------------------------- | ------------------------------------------ |
| 内容     | 三句话摘要：问题 / 原因 / 解决 | 过程日记（为什么改、怎么改的、踩了什么坑） |
| 给谁看   | 快速回顾版本差异            | 未来的自己理解当时决策                     |
| 更新时机 | 每次有可交付结果时同步      | 开发过程中随时记录                         |
| 详细程度 | 非常简洁，默认三句话        | 更详细，可写多个要点和上下文               |

### 记录时机

以下场景必须追加 Changelog：

- 交付结果改变了功能、行为、配置、构建方式、依赖、用户可感知体验
- PRD 版本升级时（随升版同步更新）
- 用户明确要求记录变更

### 顶层 Changelog 格式

日常变更（无版本号）：

```
## {YYYYMMDD} — {一句话摘要}

- 问题：{发现了什么问题或本次交付改变了什么}
- 原因：{为什么会有这个问题，或为什么需要这个变更}
- 解决：{本轮用什么方式解决，保持一句话}
```

PRD 版本升级时，在版本号旁标记版本：

```
## {YYYYMMDD} — v{x.y} — {一句话摘要}

- 问题：{版本级变更解决的核心问题}
- 原因：{为什么要升版或为什么旧方案不够}
- 解决：{本版本交付的核心解决方案}
```

### 书写规则

- 默认三条 bullet，不按“新增/修改/修复”分组
- 每条尽量一句话，不超过两行
- 不写过细的文件名、命令输出、依赖细节；这些放到 Dev-Log
- 如果用户明确要求极简，可压缩为同一段三句话，但仍保持“问题/原因/解决”的信息结构
- 日期使用 `YYYYMMDD`

### 文件位置

顶层 Changelog 放在产品总目录（与当前 `最新` PRD、Dev-Log 同级）：

```
📁 {产品总目录}/
  📄 {日期} Changelog-{产品名}.md
```

代码项目目录不再维护 Changelog 副本；PRD 版本升级时只更新产品根目录 Changelog。若发现代码目录存在历史遗留副本，先合并缺失内容，再移入产品根目录 `Archive/` 或删除，避免文档分叉。

## 七、Todo List 记录规范

### 定位

Todo List 是**任务规划清单**，记录「之后需要做但还没做的事」。
默认由用户维护，AI 不应随意写入；但当用户明确要求、指定未来版本需求，或本轮完成了 Todo 中已有任务时，AI 必须同步更新。每个产品目录下有一份独立的 Todo List 文件。

### 与其他文档的关系

| 维度               | Todo List            | PRD / Changelog / Dev-Log |
| ------------------ | -------------------- | ------------------------- |
| **更新方式** | 用户手动维护         | AI 自动更新（按 SOP）     |
| **记录内容** | 未来计划、任务优先级 | 已完成的变更和过程        |
| **触发条件** | 用户主动规划         | 代码/功能变更自动触发     |

### 推荐结构

```
# Todo List - {产品名}

## 未完成

### 🔴 高优先级
- [ ] 任务描述

### 🟡 中优先级
- [ ] 任务描述

### 🟢 低优先级
- [ ] 任务描述

## 已完成
- [x] 已完成的任务描述（完成日期：YYYY-MM-DD；结果：一句话说明） ✅
```

### 使用建议

- 新项目初始化时由 AI 创建空白 Todo List 文件，用户按需填写
- 优先级建议用 🔴 🟡 🟢 标记，一目了然
- 已完成条目应集中放在「已完成」区，保持清单可读
- AI 仅在用户明确要求、未来版本规划、或完成已有 Todo 条目时才协助整理/写入（如「帮我重新排一下优先级」「加到 v0.5」「这项完成了」）
- 如果本轮已经完成 Todo 中的未完成任务，AI 必须自动把该条从「未完成」移动到「已完成」，改为 `[x]`，补上完成日期与结果摘要，并在末尾加 `✅`；不要等用户另行提醒。

## 八、错误处理

| 异常情况             | AI 响应                                           |
| -------------------- | ------------------------------------------------- |
| 根目录无 PRD 文件    | "未找到 PRD 文件，请确认路径或创建初始版本"       |

| Changelog 不存在     | 自动创建并写入首条记录                            |
| Dev-Log 不存在       | 自动创建并写入首条记录                            |

## 九、场景速查

### 场景 A：用户说"帮我更新 PRD，加个 XX 功能"

执行「三、Skill 运作 SOP → 3.3 分支 C：完整 PRD 升版」。

### 场景 B：用户说"今天摘要"或"总结今天开发"

1. 基于当日对话，生成 Dev-Log 条目
2. 追加写入 `Dev-Log.md`

### 场景 C：用户说"归档当前 PRD"

1. 找到当前 `最新` 文件
2. 移入 `Archive/`
3. 文件名中去掉 `-最新`
4. 确认根目录无 `最新`

### 场景 D：用户说"查看当前生效的 PRD"

1. 扫描根目录 `最新` 文件
2. 0 份 → 报错："未找到当前生效 PRD"
3. 1 份 → 返回路径
4. ≥2 份 → 报错："存在多份最新文件，请人工清理"

### 场景 E：用户说"帮我整理代办"或"看看还有什么没做"

1. 读取 `Todo-List` 文件中的「未完成」条目
2. 按优先级排序输出简要清单
3. 询问用户是否需要协助调整优先级或补充条目

### 场景 F：用户说"我想开发一个新产品"或"帮我做一个 XX 应用"

执行「三、Skill 运作 SOP → 3.1 第 5 步：新产品初始化」。

## 十、首次初始化检查

如果用户首次使用此手册，执行：

1. 扫描当前目录所有 PRD 文件
2. 如果存在 `V1`、`V2` 等非标准命名 → 建议重命名为 `v0.1`、`v0.2`
3. 如果存在多份无状态标识的文件 → 询问用户哪份是当前生效版
4. 创建 `Archive/` 文件夹（如不存在）
5. 创建 `Changelog.md`（如不存在）
6. 创建 `Dev-Log.md`（如不存在）
7. 创建 `Todo-List.md`（如不存在）
8. 将非当前生效版本移入 Archive，文件名中去掉 `-最新`
9. 输出状态报告：
   ```
   当前生效 PRD：{文件名}
   最新版本：v{x.y}
   归档文件：{列表}
   Changelog：{路径}
   Dev-Log：{路径}
   Todo-List：{路径}
   ```

## 十一、安装与使用（给其他人用）

本 skill 包含一个配套的 Hook 配置文件，用于在 `git commit` 后自动提醒更新 PRD。

### 安装步骤

1. 将本 skill 文件夹放入 `~/.claude/skills/`
2. 将 `hook-settings.local.json` 复制到你的项目目录：
   ```
   cp hook-settings.local.json 你的项目/.claude/settings.local.json
   ```
3. 确保 `.gitignore` 包含 `.claude/settings.local.json`（Claude Code 默认忽略）
4. 重新打开 Claude Code，Hook 自动生效

### 验证是否生效

执行一次 `git commit`，如果看到提示：

```
📋 提示：如本次提交涉及功能变更，请更新 PRD 版本
```

则表示安装成功。
