# Vibe Coding 开发过程管理 skill

一个 Skill 管好 PRD、Changelog、Dev-Log、Todo List 四份文档，让 Vibe Coding 不再杂乱无章。

---

## 有什么问题

Vibe Coding 很爽，但聊了三小时产出一大堆代码，第二天却完全忘了为什么选这个方案。具体来说：

**需求没对齐就开写。** 跟 AI 说个大概就动手，写到一半发现理解不一致，反复改，时间全耗在返工上。

**改了什么没有记录。** 今天删了个按钮，明天换了接口，一周后复盘全靠回忆，能想起一半都算不错。

**开发过程沉底了。** 踩了什么坑、为什么选 B 方案而不是 A，对话一结束就跟着聊天记录一起丢了。未来自己打开项目，像看没有注释的代码。

**需求散落各处。** "放到 v0.5 再做""下次处理"，这些约定丢在聊天窗口里，下一次想起来已经是几个月后。

## 怎么解决：四个文档

### PRD（产品需求文档）

开发前的"对齐仪式"。你说"帮我做个新功能"，Skill 把零散需求整理成一份结构化的 PRD 草案：做什么、不做什么、怎么交互、技术选型，全列清楚。你逐条确认后再动手，从源头避免返工。

### Changelog（变更记录）

每次交付结果的摘要。改了什么、为什么改，三句话讲清楚，按时间堆叠在一个文件里。不需要翻聊天记录回忆"上周到底改了什么"，打开 Changelog 一目了然。

### Dev-Log（开发日记）

过程日记。踩坑、方案取舍、灵光一现，随时往里丢。格式不限，一两句话讲清楚上下文就行。未来自己 debug 时翻一翻，能省下几个小时。

### Todo List（待办清单）

"之后要做什么"的收容所。按优先级分层，完成了就移入「已完成」区，写清日期和结果。那些"下次再做"的约定，留在这里就不会忘记。

## 现在的开发过程变成什么样

开工前，花几分钟过一遍 PRD，确认功能范围再动手。这十分钟省下来的是后面两小时的反复修改。

过程中，AI 自动写 Dev-Log，踩坑和决策都在文件里，对话关了也不怕。交付时 Changelog 自动追加，改了什么清清楚楚。

复盘时，打开 Dev-Log 就能查到几天前踩过的坑。找文档不再靠猜，最新版永远只有一份，旧版在 Archive 里安静躺着。

与其在混乱中追求速度，不如用一点点结构换长期的清晰。
